# Big Data Technologies - Note 4 
## Advanced MapReduce: Combiners, Partitioners, In-Mapper Combining, Speculative Execution

This README summarizes the most important exam concepts from **Note 4: Advanced MapReduce**.


# 1. What is the main focus of this note?

This note focuses on **combiners**, **partitioners**, **local aggregation**, **in-mapper combining**, **algorithmic correctness**, and **speculative execution** in MapReduce.


# 2. Why are partitioners needed?

The default number of reducers is **1**, which is fine for small outputs, but for large outputs it is better to partition data across multiple reducers to use cluster parallelism.


# 3. What does a partitioner do?

A partitioner divides the intermediate key space and decides which reducer receives each intermediate key-value pair.


# 4. How many partitions does a partitioner create?

The number of partitions is the same as the number of reduce tasks.


# 5. What is the default partitioner in Hadoop?

The default partitioner is **HashPartitioner**.


# 6. How does HashPartitioner work?

It computes:

`(key.hashCode() & Integer.MAX_VALUE) % numReduceTasks`

This distributes keys approximately evenly across reducers.


# 7. When do we need a custom partitioner?

A custom partitioner is needed when we want specific keys or key ranges to go to specific reducers instead of using default hash-based distribution.


# 8. What is local aggregation?

Local aggregation is the process of reducing mapper output on the map side before shuffle, which lowers the number of intermediate key-value pairs sent over the network.


# 9. Why is local aggregation important?

Because map outputs are written to local disk and then transferred over the network, and both disk I/O and network traffic are expensive.


# 10. What is a combiner?

A combiner is a **mini-reducer** that runs after the map phase on the mapper node to aggregate local intermediate results before shuffle.


# 11. What is the main benefit of a combiner?

A combiner reduces the number of intermediate key-value pairs that must be shuffled across the network, saving bandwidth and improving performance.


# 12. On what data does a combiner operate?

A combiner operates only on the output generated by **one mapper**.


# 13. Can a reducer be used as a combiner?

Yes, but only if the reduce operation is **commutative** and **associative** and the combiner preserves correctness.


# 14. What constraints must be satisfied for combiners?

- Correctness must not depend on the combiner
- Combiner and reducer must have the same method signature
- Combiner input/output types must match mapper output and reducer input types
- The operation should be associative and commutative


# 15. Does Hadoop guarantee combiner execution?

No. Hadoop does **not** guarantee that a combiner will run, how many times it will run, or even that it will run at all.


# 16. What is the exam-safe rule about combiners?

A MapReduce job must always produce the same correct result **with or without** a combiner.


# 17. Why are combiners only an optimization?

Because they improve performance but must never change the correctness of the algorithm.


# 18. What is in-mapper combining?

In-mapper combining is a design pattern where combiner functionality is moved directly into the mapper by preserving state across multiple map calls.


# 19. How does in-mapper combining work?

The mapper stores partial aggregates in an in-memory data structure such as a `HashMap`, and emits results later instead of emitting `<key, value>` immediately for every input item.


# 20. What is Modified WordCount Version 1?

Version 1 performs local aggregation **within a single record (line)** by using a `HashMap` inside the `map()` method and emitting one pair per unique word in that line.


# 21. What is Modified WordCount Version 2?

Version 2 preserves state across multiple records in the mapper:
- initialize a map in `setup()`
- accumulate counts in `map()`
- emit final partial counts in `cleanup()`

This is the **in-mapper combiner** pattern.


# 22. Why is in-mapper combining often faster than regular combiners?

Because it gives the programmer explicit control over when and how aggregation happens, and avoids dependence on Hadoop’s optional combiner behavior.


# 23. What are advantages of in-mapper combining?

- better control over local aggregation
- often faster than regular combiners
- reduces network traffic significantly
- reduces reducer workload for frequent keys


# 24. What are disadvantages of in-mapper combining?

- preserved state can consume a lot of memory
- may introduce order-dependent bugs
- algorithm correctness can become harder to reason about


# 25. What is the memory problem with in-mapper combining?

Because the mapper stores partial aggregates in memory across many records, memory usage can grow large for high-cardinality data.


# 26. How can we reduce memory pressure in in-mapper combining?

Use **blocking and flushing**:
- process `n` input records
- emit partial results
- clear in-memory state
- continue


# 27. How does local aggregation help with skew?

It reduces the number of repeated values for high-frequency keys like `"the"`, which helps prevent reducer stragglers caused by skewed data distributions.


# 28. What is a reducer straggler?

A reducer straggler is a reducer that runs much slower than others, often because it receives disproportionately more data than the others.


# 29. Why is averaging tricky with combiners?

Because **average is not associative**, so using the reducer directly as the combiner can produce incorrect results.


# 30. Why does “average of averages” fail?

Because in general:

`Avg(1,2,3,4,5) != Avg(Avg(1,2), Avg(3,4,5))`

So averaging partial averages does not preserve correctness.


# 31. Why does Computing Average Version 1 not work well with a reducer as combiner?

Because the reducer computes averages directly, and using it as a combiner would compute averages of subsets, which produces the wrong final answer.


# 32. Why does Computing Average Version 2 also fail?

Because the combiner output type becomes a pair `(sum, count)`, but the mapper output type is still a plain integer, so the combiner types do not match the reducer expectations.


# 33. What is the correct solution for average computation?

Use the mapper to emit a pair:

`<key, (value, 1)>`

Then let both the combiner and reducer sum partial sums and partial counts, and compute the average only at the very end in the reducer.


# 34. Why does Computing Average Version 3 work?

Because it transforms a non-associative operation (average) into an associative one by aggregating **(sum, count)** pairs.


# 35. What happens if combiners do not run in Average Version 3?

The algorithm still works, because reducers can correctly aggregate all `(sum, count)` pairs directly and compute the final average.


# 36. What is the key correctness principle behind Version 3?

The algorithm remains correct no matter how many times the combiner runs - zero, one, or many - because the combiner only aggregates partial sums and counts.


# 37. Can in-mapper combining also be used for average computation?

Yes. The mapper can accumulate partial `(sum, count)` pairs in memory and emit them later, while the reducer remains the same as in the corrected version.


# 38. What is speculative execution?

Speculative execution is a technique where Hadoop launches a duplicate copy of a slow-running task on another node as a backup.


# 39. Why is speculative execution used?

Because a single slow task can delay the completion of the entire job, so Hadoop tries to hedge against slow tasks by launching duplicates.


# 40. How does speculative execution work?

- Hadoop monitors task progress
- detects tasks running much slower than average
- launches a duplicate task elsewhere
- whichever copy finishes first wins
- the duplicate is killed


# 41. Is speculative execution enabled by default?

Yes, speculative execution is enabled by default.


# 42. How can speculative execution be disabled?

It can be disabled separately for mappers and reducers through job configuration options.


# 43. What is the main point of this note?

The programmer mainly implements:
- mapper
- reducer
- optionally combiner
- optionally partitioner

and the framework handles the rest of distributed execution automatically.


# 44. Important comparison: Combiner vs In-Mapper Combiner

## Combiner
- runs after map phase
- optional
- Hadoop controls execution
- may run zero, one, or multiple times

## In-Mapper Combiner
- logic moved into mapper
- programmer controls aggregation
- usually faster
- uses mapper state and more memory


# 45. Important comparison: Default Partitioner vs Custom Partitioner

## Default Partitioner
- HashPartitioner
- uses hash(key) mod reducers
- tries to distribute keys evenly

## Custom Partitioner
- user-defined logic
- sends certain keys/ranges to chosen reducers
- useful for custom grouping or load balancing
